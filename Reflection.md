The Chapter 11 module helped me better understand how FastAPI applications separate database models from data validation. The SQLAlchemy calculation model represented the different calculator operations (e.g., addition, subtraction, etc.) in the database. This also allowed for the use of Pydantic schemas to check data prior to storing it in the database. Furthermore, the Factory pattern was used to organize the supported calculations. Additionally, the Factory pattern was useful because it made it easy to add or remove other calculation types and made the code easier to follow and maintain. Working with each of these components illustrated how each had its own responsibilities but was still able to work together as one single application.

The most difficult aspect of completing the module was ensuring that the SQLAlchemy model, Pydantic schemas, Factory classes, and the PostgreSQL database would work together properly. Additionally, I had to ensure that adding the new functionality did not negatively affect the user authentication feature developed in Module 10. Writing unit tests on the Factory and schemas, as well as writing integration tests on the database model, ensured that each individual component was tested and verified separately before testing the entire process. Also, fixing failed tests furthered my knowledge of both debugging and using automated testing methods.

Module 11 provided additional emphasis on the importance of continuous integration and deployment (CI/CD) throughout the development process. After each successful commit to the repository, GitHub Actions executed a full test suite, ran a security scan, and deployed a new Docker image. Utilizing version control through Git with frequent commits allowed me to easily see where I was in terms of progress and helped me to isolate problems as they arose. Upon completion of this module, I have a greater understanding of SQLAlchemy, Pydantic validation, automated testing methods, and CI/CD practices.
